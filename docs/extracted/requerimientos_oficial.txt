ID Actor 
Nombre  del  Requerimiento 
Descripción 
RF-1.1
 Sistema
 Autenticación  de  usuarios
 
El  sistema  requerirá  el  registro  e  inicio  de  sesión  obligatorio  para  identificar  a  los  integrantes  y  gestionar  los  permisos  de  acceso  a  la  plataforma.
 
RF-1.2
 Sistema
 Manejo  de  credenciales  inválidas
 
El  sistema  deberá  mostrar  un  mensaje  de  error  claro  cuando  las  credenciales  ingresadas  sean  incorrectas,  permitiendo  al  usuario  reintentar  el  inicio  de  sesión  sin  bloquear  la  cuenta  automáticamente.
 
RF-1.3
 Sistema
 Validación  y  enrutamiento  de  sesión  activa
 
Al  iniciar  la  aplicación,  el  sistema  deberá  verificar  internamente  si  existe  un  token  de  sesión  activo.  Si  lo  hay,  redirigirá  al  usuario  directamente  al  dashboard  o  viaje  en  curso.  Si  no,  mostrará  la  Pantalla  de  Bienvenida.
 
RF-1.4
 Viajero
 Pantalla  de  bienvenida  y  detección  de  cuenta
 
El  sistema  deberá  mostrar  una  pantalla  de  bienvenida  al  iniciar  la  aplicación  y  detectar  si  el  usuario  ya  posee  una  cuenta,  enrutándolo  al  flujo  de  inicio  de  sesión  (Sí)  o  al  flujo  de  registro  (No).
 
RF-1.5
 Viajero
 Registro  por  pasos  secuenciales
 
El  sistema  deberá  guiar  al  usuario  a  través  de  3  etapas  obligatorias  y  secuenciales:  (1)  Datos  básicos  (correo,  contraseña,  género),  (2)  Verificación  de  correo  con  código  OTP,  (3)  Configuración  de  perfil  (nickname  y  foto).  No  se  podrá  avanzar  a  la  siguiente  etapa  sin  completar  la  anterior.  La  cuenta  no  quedará  activa  hasta  completar  las  3  etapas.
 
RF-1.6
 Viajero
 Recuperación  de  contraseña
 
El  sistema  deberá  permitir  al  usuario  solicitar  el  restablecimiento  de  su  contraseña  mediante  el  correo  electrónico  registrado.  El  sistema  generará  un  enlace  de  restablecimiento  de  un  solo  uso  con  tiempo  de  expiración  de  30  minutos  y  lo  enviará  al  correo.  Al  acceder  al  enlace,  el  usuario  podrá  establecer  una  nueva  contraseña.  El  token  quedará  invalidado  permanentemente  tras  su  uso  o  expiración.
 
RF-1.7
 Viajero
 Gestión  de  perfil  personal
 
El  sistema  deberá  permitir  al  usuario  ver  y  gestionar  su  perfil  desde  una  vista  unificada  con  acceso  a:  (a)  Editar  información  personal  (nombre,  nickname,  foto,  correo),  (b)  Cambiar  contraseña,  (c)  Configurar  preferencias  de  notificación,  (d)  Consultar  sus  estadísticas  de  viaje.
 
RF-1.8
 Viajero
 Eliminación  de  cuenta  personal
 
El  sistema  deberá  permitir  al  usuario  solicitar  la  eliminación  permanente  de  su  cuenta  desde  la  Zona  de  Peligro  del  perfil.  El  proceso  requerirá  confirmación  por  correo  electrónico.  Si  el  usuario  confirma,  la  cuenta  quedará  eliminada  de  forma  permanente  y  la  sesión  será  cerrada

RNF  —  Requerimientos  No  Funcionales  
ID Categoría Requerimiento Criterio  de  Verificación 
RNF-1.1
 Seguridad
 Protección  de  datos  de  sesión:  El  sistema  no  almacenará  información  sensible  de  compras  simuladas  y  evitará  generar  logs  que  expongan  tokens  de  sesión  de  los  usuarios.
 
Auditar  los  logs  del  sistema  para  confirmar  la  ausencia  de  tokens  de  sesión.
 
RNF-1.2
 Rendimiento
 Optimización  en  memoria  con  Redis:  El  sistema  implementará  Redis  como  base  de  datos  en  memoria  para  gestionar  las  sesiones  activas  y  funcionar  como  adaptador  de  escalabilidad  para  Socket.io.
 
Monitorear  el  uso  de  memoria  de  Redis  bajo  carga  de  usuarios  concurrentes.
 
RNF-1.3
 Seguridad
 Seguridad  del  token  de  recuperación:  El  token  de  recuperación  de  contraseña  deberá  generarse  con  al  menos  256  bits  de  entropía  (crypto.randomBytes),  almacenarse  hasheado  en  PostgreSQL  y  tener  una  vigencia  máxima  de  30  minutos.  Una  vez  utilizado  o  expirado,  el  token  debe  invalidarse  permanentemente.
 
Intentar  reutilizar  un  token  ya  consumido  y  verificar  rechazo.  Intentar  usarlo  después  de  30  minutos  y  verificar  invalidación.
 
RNF-1.4
 Disponibilidad
 
Servicio  de  correo  transaccional:  El  sistema  deberá  integrarse  con  Nodemailer+SMTP  o  SendGrid  para  el  envío  del  enlace  de  recuperación.  El  correo  deberá  llegar  al  usuario  en  un  máximo  de  60  segundos  bajo  condiciones  normales.
 
Solicitar  recuperación  con  correo  válido  y  medir  el  tiempo  de  recepción.  Verificar  que  llegue  en  menos  de  60  segundos.
 
RNF-1.5
 Seguridad
 Verificación  de  correo  con  código  OTP:  El  código  OTP  enviado  durante  el  registro  deberá  tener  una  vigencia  máxima  de  10  minutos  y  ser  de  un  solo  uso.  Si  expira  antes  de  ser  validado,  el  sistema  deberá  permitir  al  usuario  solicitar  el  reenvío  de  un  nuevo  código  sin  reiniciar  el  flujo  de  registro  completo.
 
Intentar  usar  un  código  OTP  después  de  10  minutos  y  verificar  rechazo.  Verificar  que  el  reenvío  genere  un  nuevo  código  válido  sin  reiniciar  el  registro.
 
RNF-1.6
 Seguridad
 Invalidación  de  sesiones  al  eliminar  cuenta:  Al  completarse  la  eliminación  de  una  cuenta,  todos  los  tokens  de  sesión  activos  del  usuario  deberán  ser  revocados  inmediatamente  en  Redis,  impidiendo  cualquier  acceso  posterior  con  credenciales  o  tokens  previos.
 
Eliminar  una  cuenta  con  sesión  activa  y  verificar  que  el  token  quede  invalidado  en  Redis.  Intentar  usar  el  token  revocado  y  confirmar  el  rechazo.

RNB  —  Reglas  de  Negocio  
ID Tipo Regla  de  Negocio 
RNB-1.1
 Restricción
 Asignación  del  rol  base  en  el  registro:  Al  completar  el  flujo  de  registro,  el  sistema  asigna  automáticamente  el  rol  base  de  Viajero.  La  asignación  del  rol  de  Administrador  de  Grupo  ocurre  en  el  Módulo  2,  cuando  el  usuario  crea  un  nuevo  espacio  de  viaje,  y  no  forma  parte  del  flujo  de  autenticación.
 
RNB-1.2
 Restricción
 Flujo  de  registro  secuencial  e  irrenunciable:  El  usuario  no  puede  omitir  ninguna  de  las  3  etapas  del  flujo  de  registro.  Si  abandona  el  flujo  antes  de  completarlo,  la  cuenta  no  queda  activa.  Los  datos  parciales  podrán  conservarse  por  un  máximo  de  24  horas  para  permitir  retomar  el  proceso;  transcurrido  ese  tiempo  serán  descartados.
 
RNB-1.3
 Restricción
 Validez  del  token  de  recuperación:  El  enlace  de  recuperación  tiene  una  vigencia  máxima  de  30  minutos  desde  su  generación.  Un  token  solo  puede  ser  utilizado  una  única  vez;  tras  su  uso  exitoso,  queda  invalidado  permanentemente  en  la  base  de  datos.
 
RNB-1.4
 Restricción
 Correo  registrado  obligatorio  para  recuperación:  El  sistema  solo  puede  enviar  el  enlace  de  recuperación  a  correos  electrónicos  asociados  a  una  cuenta  activa  y  verificada.  Si  el  correo  no  existe  o  la  cuenta  está  inactiva,  el  sistema  mostrará  un  mensaje  genérico  sin  confirmar  ni  negar  la  existencia  de  la  cuenta,  para  evitar  la  enumeración  de  usuarios.
 
RNB-1.5
 Restricción
 Formato  de  nueva  contraseña:  La  nueva  contraseña  deberá  cumplir  los  requisitos  mínimos  de  seguridad:  mínimo  8  caracteres,  al  menos  una  mayúscula,  una  minúscula  y  un  número.  No  podrá  ser  idéntica  a  la  contraseña  anterior  almacenada.
 
RNB-1.6
 Restricción
 Formato  obligatorio  del  nickname:  El  nickname  del  usuario  debe  ser  único  en  el  sistema  y  no  puede  contener  caracteres  especiales  distintos  a  guion  bajo  (_)  y  punto  (.).  La  foto  de  perfil  es  opcional  durante  el  registro,  pero  el  nickname  es  obligatorio  para  completar  la  etapa  3  y  activar  la  cuenta.
 
RNB-1.7
 Restricción
 Eliminación  de  cuenta  irreversible:  Una  vez  que  el  usuario  confirma  la  eliminación  de  su  cuenta  a  través  del  correo  de  verificación,  la  operación  no  puede  deshacerse  bajo  ninguna  circunstancia.  El  sistema  no  debe  ofrecer  ninguna  opción  de  recuperación  posterior.

ID Actor 
Nombre  del  Requerimiento 
Descripción 
RF-2.1
 Sistema
 Detección  y  reanudación  de  viaje  activo
 
El  Selector  de  Viajes  deberá  bifurcar  la  navegación  en  tres  caminos:  (a)  Continuar  viaje  en  curso,  retomando  estado  y  rol  previo  sin  reconfiguración;  (b)  Crear  Nuevo  Viaje;  (c)  Unirse  a  un  viaje  existente.  Si  el  usuario  ya  tiene  un  viaje  activo,  el  sistema  cargará  automáticamente  el  rol  y  permisos  vigentes  sin  solicitar  ninguna  acción  adicional.
 
RF-2.2
 Sistema
 Asignación  y  carga  de  permisos  por  rol
 
Al  cargar  el  dashboard  de  viaje  activo,  el  sistema  deberá  determinar  el  rol  del  usuario  dentro  de  ese  viaje  específico  (Admin  o  Viajero)  basándose  en  cómo  ingresó  al  viaje,  y  cargar  de  forma  reactiva  el  conjunto  de  permisos  correspondiente.
 
RF-2.3
 Admin
 Permisos  exclusivos  del  rol  Admin
 
El  rol  Admin  dentro  de  un  viaje  deberá  tener  acceso  exclusivo  a:  (a)  Gestionar  integrantes:  invitar,  expulsar,  ver  perfiles;  (b)  Bloquear  y  desbloquear  el  módulo  de  Itinerario;  (c)  Bloquear  y  desbloquear  el  módulo  de  Presupuesto;  (d)  Eliminar  reservas  de  la  Bóveda;  (e)  Delegar  el  rol  de  Admin  a  otro  Viajero;  (f)  Cerrar  el  viaje  (archivar  en  historial);  (g)  Emitir  Decisiones  Administrativas  (override  sin  votación);  (h)  Confirmar,  bloquear  o  desempatar  propuestas  grupales.
 
RF-2.4
 Viajero
 Permisos  del  rol  Viajero
 
El  rol  Viajero  dentro  de  un  viaje  deberá  tener  acceso  a:  (a)  Ver  actividades  del  itinerario;  (b)  Crear  propuestas  grupales;  (c)  Votar  en  propuestas  activas;  (d)  Ver  sus  finanzas  personales;  (e)  Participar  en  los  chats  del  viaje;  (f)  Proponer  la  división  en  subgrupos.
 
RF-2.5
 Usuario  (Admin)
 
Crear  nuevo  viaje
 El  sistema  deberá  permitir  al  usuario  crear  un  viaje  nuevo  ingresando  nombre,  descripción,  temporalidad  y  presupuesto  base.  Al  crearlo,  el  sistema  asignará  automáticamente  el  rol  Admin  al  creador.
 
RF-2.6
 Usuario
 Unirse  a  un  viaje
 El  sistema  deberá  permitir  al  usuario  ingresar  a  un  viaje  existente  mediante  un  código,  enlace  QR  o  invitación  por  correo.  Al  unirse,  el  sistema  asignará  el  rol  Viajero.
 
RF-2.7
 Usuario
 Consultar  historial  de  viajes
 
El  sistema  deberá  permitir  al  usuario  visualizar  la  lista  de  sus  viajes  anteriores  y  actuales,  siempre  accesible  desde  el  Selector  de  Viajes.
 
RF-2.8
 Usuario  (Admin)
 
Invitar  integrantes
 El  sistema  deberá  permitir  al  Admin  generar  y  compartir  un  enlace,  código  QR  o  invitación  por  correo  para  sumar  integrantes.  El  Admin  podrá  configurar  una  fecha  de  expiración  para  el  enlace  o  un  límite  máximo  de  usos.

Cerrar  viaje
 El  Admin  podrá  archivar  el  viaje  activo,  moviéndolo  al  historial.  Una  vez  cerrado,  el  viaje  quedará  en  modo  solo  lectura  y  no  podrá  reactivarse.

 Usuario

RF-2.13
 Sistema

RNF  —  Requerimientos  No  Funcionales 
ID Categoría Requerimiento Criterio  de  Verificación 
RNF-2.1
 Seguridad
 Persistencia  de  rol  aislada  por  viaje:  El  rol  de  un  usuario  (Admin  o  Viajero)  es  específico  a  cada  viaje  y  se  persiste  entre  sesiones  de  forma  aislada.  Un  usuario  puede  ser  Admin  en  un  viaje  y  Viajero  en  otro  simultáneamente.
 
Verificar  que  al  cambiar  de  viaje  el  sistema  cargue  el  rol  correcto  para  ese  contexto  específico.
 
RNF-2.2
 Rendimiento
 Carga  reactiva  de  permisos:  Los  permisos  del  rol  deben  cargarse  y  reflejarse  en  la  UI  en  menos  de  500ms  tras  la  validación  del  token  de  sesión  del  viaje.
 
Medir  el  tiempo  desde  la  validación  del  token  hasta  la  renderización  de  los  controles  de  rol  en  el  dashboard.
 
RNF-2.3

RF-5.1
 Sistema
 Conexión  WebSocket  con  validación  JWT
 
El  sistema  deberá  establecer  la  conexión  WebSocket  realizando  el  handshake  inicial  con  validación  del  token  JWT  del  usuario.  Una  vez  validada,  asignará  la  conexión  a  la  sala  correspondiente  (trip_id)  para  aislar  los  eventos  del  viaje.
 
RF-5.2
 Sistema
 Heartbeat  de  conectividad
 
El  sistema  deberá  emitir  un  pulso  (ping)  al  servidor  cada  30  segundos  para  confirmar  que  el  dispositivo  del  cliente  mantiene  conexión  activa.  Si  no  se  recibe  respuesta  en  30  segundos,  el  socket  deberá  declararse  inactivo.
 
RF-5.3
 Sistema
 Modo  offline  con  bloqueo  de  escritura
 
El  Network  Monitor  deberá  detectar  la  pérdida  de  conectividad  mediante  el  heartbeat.  Ante  la  pérdida  de  conexión,  el  sistema  deberá:  (a)  Activar  inmediatamente  el  Modo  Lectura,  bloqueando  todos  los  botones  y  acciones  de  escritura  en  la  UI;  (b)  Mostrar  un  banner  de  aviso  persistente  al  usuario  indicando  'Sin  conexión  —  modo  lectura  activo';  (c)  Almacenar  localmente  mediante  IndexedDB/Caché  los  cambios  que  el  usuario  intente  realizar;  (d)  Encolar  dichos  cambios  en  una  Cola  de  Sincronización  Diferida  para  enviarlos  al  servidor  una  vez  restaurada  la  conexión.
 
RF-5.4
 Sistema
 Reincorporación  de  sesión  y  vaciado  de  cola
 
Al  restaurarse  la  conexión,  el  sistema  deberá  reconectar  el  socket,  solicitar  los  eventos  perdidos  durante  la  desconexión  y  vaciar  la  Cola  de  Sincronización  Diferida.  El  tiempo  límite  para  una  reconexión  rápida  sin  recargar  la  página  completa  es  de  60  segundos.
 
RF-5.5
 Sistema
 Resolución  de  conflictos  de  sincronización
 
Al  vaciar  la  Cola  de  Sincronización  Diferida,  el  sistema  deberá:  (a)  Detectar  si  existen  conflictos  entre  los  cambios  locales  pendientes  y  el  estado  actual  del  servidor;  (b)  Si  no  hay  conflicto:  ejecutar  merge  automático  sin  superposición  de  campos;  (c)  Si  hay  conflicto:  aplicar  la  política  Last  Write  Wins  (la  última  escritura  en  el  servidor  gana)  y  notificar  al  usuario  afectado  que  su  cambio  fue  sobreescrito.
 
RF-5.6
 Sistema
 Difusión  de  actualizaciones  de  presencia
 
El  sistema  deberá  emitir  los  eventos  USER_ONLINE  y  USER_OFFLINE  a  todos  los  integrantes  de  la  sala  del  viaje  cuando  un  usuario  se  conecte  o  desconecte.

 Sistema
